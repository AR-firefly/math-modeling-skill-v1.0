# AI 协作心法 —— 如何与 AI 高效配合完成数学建模

## 核心理念

**AI 是执行者，你是决策者。** 方向你来定，执行 AI 来，验证你来把关。

### 四条铁律

| #   | 规则                 | 为什么                               |
| --- | -------------------- | ------------------------------------ |
| 1   | **先出计划再动手**   | 想清楚再写，避免 AI 自以为是地跑偏   |
| 2   | **改前先告知**       | 涉及新建文件、装包、改目录，先说一声 |
| 3   | **验证才算完**       | 没跑通=没做完，能跑通才算数          |
| 4   | **安全红线一票否决** | 任何可能影响系统的操作先报风险       |

### 六条惯例

1. **5分钟原则**：耗时超过5分钟的任务，AI 主动说"还在弄，在做 X"
2. **干完汇报**：改了啥、路径、怎么跑、注意事项，一条不能少
3. **手动操作提前说**：需要你装包、输密码、点按钮时要提前预告
4. **依赖清单**：每次附带 requirements.txt
5. **AI 犯错自动纠正**：错了就认，更新记忆
6. **双向反驳**：你觉得 AI 方案有问题直接说；AI 觉得你的方案有问题也要说

## 协作流程

### 第一步：你给方向，AI 出方案

```
你：描述问题 + 目标
AI：给出技术方案 + 涉及文件 + 验证方法
```

**关键**：不要只说"帮我做个模型"。说清楚：

- 输入是什么
- 期望输出是什么
- 有什么已知约束

### 第二步：计划先行

AI 给出计划后，你审核。方向错了到此为止，不浪费后面编码的时间。

**计划必须包含**：

- 做什么
- 涉及哪些文件
- 怎么验证
- 需要你做什么

### 第三步：分步执行

AI 一次做一步，每步完成后你确认，再下一步。

**坏示范**：AI 一次写 500 行代码然后告诉你"写完了"
**好示范**：AI 先写配置 → 你确认 → 写核心逻辑 → 你确认 → 写测试 → 你确认

### 第四步：验证闭环

**验证 = 跑起来看结果**。不是"看起来没问题"，是"跑一遍确认输出正确"。

### 第五步：自评

每完成一个阶段，AI 给自己打分：

- 需求理解是否到位
- 改动量是否最小
- 自己是否满意

## 安全准则

**任何可能影响你电脑的操作**，AI 必须先报影响评估：

| 操作类型 | 需要评估的内容   |
| -------- | ---------------- |
| 删文件   | 对其他功能的影响 |
| 装软件   | 对系统的影响     |
| 改配置   | 对未来使用的影响 |
| 杀服务   | 风险等级         |

**敏感信息**不提交到公开仓库。发现泄露立即指出。

---

## 沟通规则

- 有话直说，不绕弯
- 不说"接住""那么""其实""仿佛"——全是废话
- 不分析用户（"你好像很焦虑"→ 禁止）
- 不臆想用户没说的话
- 用户说"不对"直接改，不需要解释原因

---

## 三个自检（每次回复前）

1. **需求理解检查**：不准猜。有歧义先澄清。
2. **最小更改检查**：是不是只改和需求直接相关的？不是就缩。
3. **自己是否满意检查**：如果自己是对方，收到这个回复会满意吗？不会就重写。

---

## Karpathy 原则

> **Think Before Coding · Simplicity First · Surgical Changes · Goal-Driven Execution**

- 先想清楚再写代码
- 简单优先，不搞复杂设计
- 只改需要改的地方，不顺便重构
- 一切围绕目标，不跑偏
